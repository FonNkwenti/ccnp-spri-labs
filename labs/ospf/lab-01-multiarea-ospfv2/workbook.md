## 1. Concepts & Skills Covered

**Exam Objective:** Blueprint bullets 1.2 and 1.2.a вҖ” Troubleshoot OSPF multiarea operations and route advertisement across backbone and non-backbone areas.

This lab transitions you from single-area OSPF (lab-00) into a realistic multiarea topology where routing information crosses area boundaries via Area Border Routers (ABRs). You will observe how OSPF's LSA types change as routes propagate between areas, understand the role of the OSPF backbone (Area 0), and verify end-to-end reachability across a multi-domain network.

### Area Architecture and ABRs

OSPF divides large domains into areas to reduce CPU overhead and LSA flooding. Area 0 is the backbone and must be present; all non-backbone areas connect through it. An Area Border Router (ABR) belongs to multiple areas simultaneously and originates Type-3 (Summary) LSAs to advertise inter-area routes. In this lab, R2 is an ABR for Areas 0 and 1, while R3 is an ABR for Areas 0, 2, and 3.

### LSA Types and Inter-Area Propagation

- **Type-1 (Router LSA):** Flooded only within the originating area; describes a router's intra-area links.
- **Type-2 (Network LSA):** Generated by the designated router on a multi-access segment; flooded within the area.
- **Type-3 (Summary LSA):** Generated by ABRs to advertise intra-area routes from one area into another. An ABR generates a Type-3 for each subnet it learns within its connected areas.

When a router in Area 1 (e.g., R1) learns about a subnet in Area 2 (e.g., 172.16.4.0/24), the path is: Type-1 LSA (R4 in Area 2) вҶ’ Type-3 LSA (R3 to Area 0) вҶ’ Type-3 LSA (R2 to Area 1) вҶ’ R1 receives Type-3, calculates inter-area route via R2.

### Multiarea Design Principles

- Area 0 connects all other areas and must be contiguous (no Area-0-to-Area-0 links through other areas).
- Keep Area 0 small (few routers, few intra-area routes) to minimize SPF computation overhead.
- Non-backbone areas can be isolated by ABRs вҖ” a failure within Area 2 does not affect Area 3 intra-area SPF.
- ABRs filter and originate Type-3 LSAs; advanced features (area range, stub areas) are configured on ABRs.

### Equivalence and Differences from IS-IS

OSPFv2's area model is similar to IS-IS's L1/L2 model: both use level 1 (area-internal) and level 2 (backbone) to reduce flooding. However, OSPF's Type-3 Summary LSA explicitly lists each summarized subnet, while IS-IS's DIS on L2 floods the entire L2 topology. This lab anchors the OSPF comparison from lab-00; the IS-IS topic carries a cross-reference rather than re-teaching multiarea concepts.

## 7. Verification Cheatsheet

### OSPF Process Configuration

```
router ospf 1
 router-id 10.0.0.N
 network [subnet] [wildcard] area [N]
```

| Command | Purpose |
|---------|---------|
| `router ospf 1` | Enable OSPF process 1 |
| `router-id 10.0.0.N` | Set unique router identifier (typically loopback IP) |
| `network 10.1.12.0 0.0.0.255 area 1` | Advertise the 10.1.12.0/24 subnet into Area 1 |

> **Exam tip:** Always configure `router-id` explicitly to avoid router-id changes during network operations.

### Area Configuration

| Device | Area 0 | Area 1 | Area 2 | Area 3 |
|--------|--------|--------|--------|--------|
| R1 | вҖ” | Lo0, Lo1, Gi0/0 | вҖ” | вҖ” |
| R2 | Lo0, Gi0/1 | Gi0/0 | вҖ” | вҖ” |
| R3 | Lo0, Gi0/0 | вҖ” | Gi0/1 | Gi0/2 |
| R4 | вҖ” | вҖ” | Lo0, Lo1, Gi0/0 | вҖ” |
| R5 | вҖ” | вҖ” | вҖ” | Lo0, Lo1, Gi0/0 |

> **Exam tip:** ABRs (R2, R3) connect multiple areas; ensure each interface is assigned to exactly one area.

### Verification Commands

| Command | What to Look For |
|---------|-----------------|
| `show ip ospf neighbor` | All neighbors in FULL state |
| `show ip ospf interface brief` | Correct area assignment per interface |
| `show ip ospf database \| include "Type-3"` | Type-3 LSAs from R2 (R2 is originator) and R3 |
| `show ip route ospf` | Inter-area routes marked `O IA` |
| `ping 172.16.4.1 repeat 100` | End-to-end reachability test |

### Wildcard Mask Quick Reference

| Subnet Mask | Wildcard Mask | Common Use |
|-------------|---------------|------------|
| 255.255.255.255 | 0.0.0.0 | Single host (/32) |
| 255.255.255.0 | 0.0.0.255 | /24 network (OSPF links, loopbacks) |
| 255.255.0.0 | 0.0.255.255 | /16 network |

### Common OSPF Multiarea Failure Causes

| Symptom | Likely Cause |
|---------|-------------|
| Neighbors stuck in ExStart | Mismatched area on one side of the link; mismatched hello/dead timers |
| No Type-3 LSAs in database | ABR not configured correctly; area not enabled on ABR's interface |
| No inter-area routes in routing table | Type-3 received but suppressed by `area X filter-list`; metric too high |
| High inter-area latency | Inter-area path longer than expected; check `show ip ospf database summary` for metric |

### Task 5 Solution: Diagnosing Area-Mismatch Fault

<details>
<summary>Click to view Diagnosis Steps</summary>

1. **Observe symptom:** R3 and R4 neighbors in ExStart state.
   ```bash
   show ip ospf neighbor detail
   ```
   Look for ExStart, Exchange, or Loading state between R3 and R4.

2. **Check area configuration:**
   ```bash
   show ip ospf interface Gi0/1
   show run | include "network 10.1.34"
   ```
   Verify that 10.1.34.0/24 is assigned to Area 2 on both R3 and R4.

3. **Identify mismatch:**
   If R3 Gi0/1 shows `Area 0` but should be `Area 2`, the issue is found.

</details>

<details>
<summary>Click to view Fix</summary>

```bash
R3# configure terminal
R3(config)# router ospf 1
R3(config-router)# no network 10.1.34.0 0.0.0.255 area 0
R3(config-router)# network 10.1.34.0 0.0.0.255 area 2
R3(config-router)# end
```

After fix, `show ip ospf neighbor` will show R3-R4 as Full; routes toward R4 return to R1's routing table.
