## 1. Concepts & Skills Covered

**Exam Objective:** 1.1 — Describe OSPF operations and compare with IS-IS

This lab establishes the OSPFv2 baseline for the entire OSPF topic. Three routers are
brought up in a single backbone area, giving you a clean environment to examine the
adjacency state machine, the link-state database, and LSA generation before the
topology grows in complexity in labs 01–03. It also anchors the key conceptual
differences between OSPF and IS-IS — a distinction that appears directly on the 300-510
blueprint and recurs throughout the SP routing topics.

### OSPF Adjacency Formation — The Seven States

OSPF neighbours do not exchange routing information until they reach the FULL state. The
path from DOWN to FULL is a finite-state machine with seven steps:

| State | What Happens |
|-------|-------------|
| **DOWN** | No Hellos received yet; the neighbour entry does not exist |
| **INIT** | A Hello was received, but the local router ID is not yet in the neighbour's Hello |
| **2-WAY** | Both sides see each other's RID in the Hello; DR/BDR election happens here |
| **EXSTART** | Master/slave negotiation begins using empty DBD packets; higher RID wins master |
| **EXCHANGE** | Full DBD packets (list of LSA headers) are exchanged |
| **LOADING** | LSR/LSU/LSAck: missing LSAs are requested and delivered |
| **FULL** | LSDB is synchronised; routing table can be computed |

On broadcast segments (Ethernet), routers only reach FULL with the DR and BDR. All other
routers stop at 2-WAY with each other — this is normal and expected.

### LSA Types 1 and 2 — The Single-Area Database

In a single area, only two LSA types are present:

**Type 1 — Router-LSA:**
- Generated by every OSPF router in the area
- Describes the router's directly connected links and their states (transit, stub, etc.)
- Flooded only within the originating area
- Identified by: Link State ID = originating router's RID

**Type 2 — Network-LSA:**
- Generated by the DR on each multi-access segment (Ethernet)
- Describes which routers are attached to that segment
- Only present when a DR exists (not on point-to-point links)
- Identified by: Link State ID = DR's interface IP on that segment

```
show ip ospf database router    ! Type 1 — one entry per router
show ip ospf database network   ! Type 2 — one entry per DR interface
```

### OSPF vs IS-IS — Architectural Comparison

The 300-510 blueprint explicitly requires the ability to compare OSPF and IS-IS. The
table below captures the most exam-relevant differences:

| Dimension | OSPF (this lab) | IS-IS (isis topic) |
|-----------|-----------------|---------------------|
| PDU encapsulation | IP protocol 89 | Directly over CLNS (ISO 10589) |
| Topology unit | LSA | LSP (Link State PDU) |
| Area boundary router | ABR (configures per-interface area) | L1/L2 router (is-type) |
| External routing | ASBR generates Type-5 (or Type-7 NSSA) | External TLV in LSP |
| Inter-area route summary | ABR uses `area N range` | L1/L2 router uses `summary-address` |
| Protocol independence | IPv4-only in OSPFv2; IPv6 needs OSPFv3 | Multi-topology carries both in one process |
| SP deployment preference | Common on enterprise and IOS-XE PE | Preferred in SP cores (protocol-independent) |

The key insight: IS-IS runs **below** IP, which makes it immune to IP-layer
misconfigurations (wrong IP, missing route) that would break an OSPF adjacency. This is
why most SP operators run IS-IS as the IGP and use OSPF only at the CE or enterprise edge.

## 6. Verification & Analysis

### Adjacency State

```
R2# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
10.0.0.1          1   FULL/DR         00:00:38    10.1.12.1       GigabitEthernet0/0
10.0.0.3          1   FULL/BDR        00:00:34    10.1.23.3       GigabitEthernet0/1
! ← Both neighbours in FULL state — LSDB is synchronised
! ← R2 is BDR on Gi0/1 (R3's RID 10.0.0.3 > R2's 10.0.0.2, so R3 is DR)
! ← R1 is DR on Gi0/0 (R1's RID 10.0.0.1 < R2's 10.0.0.2, so R2 is DR — check this)
```

> **Note:** DR election uses highest priority (default 1 = tie) → then highest router-ID.
> On Gi0/0 (R1–R2): R2's RID 10.0.0.2 > R1's 10.0.0.1 → **R2 is DR, R1 is BDR**.
> On Gi0/1 (R2–R3): R3's RID 10.0.0.3 > R2's 10.0.0.2 → **R3 is DR, R2 is BDR**.

## 7. Verification Cheatsheet

### OSPFv2 Process Configuration

```
router ospf 1
 router-id <loopback0-ip>
 network <subnet> <wildcard> area <N>
```

| Command | Purpose |
|---------|---------|
| `router ospf 1` | Start OSPF process 1 (process ID is locally significant) |
| `router-id 10.0.0.X` | Pin the router-ID to Loopback0; prevents instability if an interface goes down |
| `network A.B.C.D W.X.Y.Z area N` | Match interfaces by IP range; advertise matched networks into the specified area |

> **Exam tip:** `router-id` takes effect immediately on a new process but requires `clear ip ospf process` on a running process. Always set it explicitly — never rely on IOS auto-selection.

### Interface Timer Controls

```
interface GigabitEthernet0/0
 ip ospf hello-interval <seconds>
 ip ospf dead-interval <seconds>
```

| Command | Purpose |
|---------|---------|
| `ip ospf hello-interval 3` | Set hello to 3s (must match neighbour) |
| `ip ospf dead-interval 12` | Set dead to 12s (must be ≥ 4× hello on both sides) |
| `no ip ospf hello-interval` | Restore default hello (10s on Ethernet) |
| `no ip ospf dead-interval` | Restore default dead (40s on Ethernet) |

> **Exam tip:** Timer mismatch is one of the most common OSPF troubleshooting scenarios. Mismatched timers cause the adjacency to drop *after* the dead interval — not immediately. Use `show ip ospf interface` to compare both sides.

### Verification Commands

| Command | What to Look For |
|---------|-----------------|
| `show ip ospf neighbor` | State must be FULL; DR/BDR roles visible |
| `show ip ospf neighbor detail` | Hello/Dead timer values, neighbour uptime |
| `show ip ospf database` | Count of LSA types; all three router-IDs present |
| `show ip ospf database router` | Type 1 LSA details per router |
| `show ip ospf database network` | Type 2 LSA; Link State ID = DR's interface IP |
| `show ip ospf interface Gi0/0` | DR/BDR election result, hello/dead timers, area |
| `show ip route ospf` | OSPF-learned routes (O prefix, AD=110) |
| `ping X.X.X.X source Lo0` | End-to-end loopback reachability |

### Wildcard Mask Quick Reference

| Subnet | Wildcard Mask | Matches |
|--------|---------------|---------|
| /32 host | 0.0.0.0 | Exactly one IP address |
| /24 | 0.0.0.255 | 256 addresses in the /24 |
| /30 | 0.0.0.3 | 4 addresses in the /30 |
| /16 | 0.255.255.255 | Entire /16 |

### Common OSPF Failure Causes

| Symptom | Likely Cause |
|---------|-------------|
| Stuck in INIT | Neighbour receives Hellos but local RID is not echoed back — check ACLs, one-way reachability |
| Stuck in EXSTART/EXCHANGE | MTU mismatch between neighbours |
| Neighbour FULL but route missing | Missing `network` statement; check `show ip ospf interface` |
| Adjacency flapping every dead-interval | Timer mismatch — compare both sides with `show ip ospf interface` |
| Type 2 LSA missing | No DR on segment (point-to-point network type) |

### Ticket 1 — R1 Loses Its Neighbour Relationship with R2

The NOC has received an alert that OSPF is flapping on the R1–R2 link. R1 and R2 were
fully adjacent a few minutes ago. The only recent change was a timer optimisation

**Inject:** `python3 scripts/fault-injection/inject_scenario_01.py --host <eve-ng-ip>`

**Success criteria:** `show ip ospf neighbor` on R2 shows R1 in FULL state with default

<details>
<summary>Click to view Diagnosis Steps</summary>

! R1 may be missing or stuck in INIT/EXSTART

R1# show ip ospf interface GigabitEthernet0/0
! Look at Hello: and Dead: values — compare with R2's

R2# show ip ospf interface GigabitEthernet0/0
! Mismatch in hello-interval or dead-interval causes adjacency to drop
! after the dead-interval expires (up to 40s after the change)

R1# show running-config | section ospf
! Check for ip ospf hello-interval or ip ospf dead-interval under the interface

</details>

Following a maintenance window last night, the operations team reports that 10.0.0.3/32
